## Problem Statement

新入职一家做 iOS 订阅工具类 App 的公司（App 工厂模式）。公司通过投流获客，主要产品是 AI 识图类工具（植物扫描、鸟类扫描等），采用周订阅 + 付费墙模式。需要快速融入团队，用 AI 能力加速 PRD 编写和原型设计两个核心瓶颈环节，同时建立长期的 AI 驱动产品开发流程。

## Demand Evidence

- 公司明确看重 AI 设计 PRD 和原型的能力（招聘/入职时的核心期望）
- App 工厂模式的核心竞争力是速度——谁能更快发现热点、更快上线、更快验证 ROI
- 当前从设计到上线周期 1-2 个月，PRD 和 UI/交互设计是占用时间最长的环节
- 产品池子模式意味着产品选择和设计效率直接决定公司产出数量

## Status Quo

**当前工作流：**
1. 从产品池子中挑选 App 方向
2. PM 用传统方式编写 PRD（功能清单 + 具体实现事项）
3. 设计师在 Figma 中制作原型（1 UI + 1 交互设计师）
4. 2 名 iOS 开发 + 1 名后端开发实现
5. 上线后投流验证

**当前痛点：**
- PRD 编写耗时较长，需要手动调研、手动撰写
- 从 PRD 到 Figma 原型有设计师介入的等待时间
- 产品结构高度相似（扫描→识别→结果→付费墙→订阅），但每次仍从零设计

**团队规模（6人）：** 1 PM + 1 UI + 1 交互 + 2 iOS + 1 后端

## Target User & Narrowest Wedge

**目标用户：** 你自己——一个需要快速证明价值的新入职产品经理

**最窄切入点：** 用 AI 在 2-3 天内产出第一个完整的、符合团队标准的 PRD + Figma 原型（而非传统的 2-4 周），以此建立信任和证明 AI 工作方式的可行性。

## Constraints

1. PRD 必须符合团队现有的质量标准和格式规范
2. 原型工具是 Figma（团队已在使用，不能更换）
3. 需要先了解现有产品的结构和标准，才能产出有效的模板
4. 团队对 AI 工具的接受程度未知，需要渐进式引入
5. 产品类型高度同质化——核心流程都是"扫描→识别→结果→付费墙"

## Premises

1. **首要目标是"先达标"** — PRD 和原型必须符合团队现有的质量标准和格式规范，不能因为用了 AI 就变成团队看不懂的东西
2. **产品模式高度相似** — 不同 App 之间主要差异在于识别对象和内容呈现，而非交互模式
3. **速度比完美更重要** — 80 分但两周上线的 App 比 95 分但两个月上线的 App 对公司更有价值
4. **最大风险是不了解团队标准** — 如果产出的 PRD 格式和团队预期不一致，再快也没用

## Approaches Considered

### Approach A: "模板化加速器" (Template-First Accelerator)

先拆解现有成功 App 的 PRD 和 Figma 原型，提取通用模板，然后用 Claude 快速填充新场景的差异化内容。

- **Effort:** M（1-2 周建模板，之后每个新 App 2-3 天出 PRD+原型）
- **Risk:** Low
- **Pros:**
  - 产出格式 100% 兼容团队现有标准
  - 模板建好后，新产品产出速度从 2-4 周缩短到 2-3 天
  - 入职第一周就能开始产出
- **Cons:**
  - 初始建模板需要研究现有产品
  - 只适用于结构类似的识图类 App

### Approach B: "AI 全栈 PM" (AI-Powered Full-Stack PM)

建立完整的 AI 辅助产品工作流——用 Claude 写 PRD，用 Figma Make 出原型，用 Claude Code 做技术可行性验证。

- **Effort:** L（3-4 周搭建工作流，长期回报最大）
- **Risk:** Medium
- **Pros:**
  - 长期把产品周期从 1-2 月压缩到 2-3 周
  - 辅助竞品分析、数据分析等全链路
  - 最能体现"AI 能力"差异化价值
- **Cons:**
  - 学习曲线较陡
  - 需要团队配合适应新协作方式

### Approach C: "先融入，后改造" (Embed-Then-Transform)

第一个月完全按团队现有方式工作，第二个月展示 AI 能力，第三个月推动流程升级。

- **Effort:** S
- **Risk:** Low
- **Pros:** 最安全的入职策略
- **Cons:** 最慢体现 AI 价值，可能错过窗口期

## Recommended Approach

**A+B+C 融合：先理解团队标准（C 的精华），用模板化加速器快速产出（A），同时渐进建设 AI 全栈 PM 能力（B）**

> 注：推荐方案本质是 A+B，但借鉴了 Approach C 的"先理解后行动"理念——第 1 周专注于理解团队标准，而非急于交付。

### Day 1 前置验证（入职当天）

在开始任何工作之前，先确认以下关键前提：
- [ ] **工具权限：** 确认公司 Figma 账号计划等级（是否支持 Figma Make / AI 功能）
- [ ] **AI 工具合规：** 确认公司是否允许使用外部 AI 工具处理产品数据（安全/合规检查）
- [ ] **Claude 访问：** 确认能否使用 Claude Pro/Team（是否需要采购审批）
- [ ] **资料获取：** 向 leader 申请获取 2-3 个已上线 App 的完整 PRD + Figma 源文件

> **工具成本预估：** Claude Pro ~$20/月，Claude Team ~$30/人/月。Figma Make 包含在 Figma 付费计划中。如果公司 Figma 账号不支持 AI 功能，手动 Figma 设计是可行的替代方案。

> **如果 AI 工具不可用的降级方案：** PRD 仍可用 Claude 网页版辅助写作（不涉及公司数据上传）；原型设计回退到传统 Figma 手动设计，但利用模板组件库加速。

### 执行路线图

#### 第 1-2 周：逆向工程 + 模板构建（Approach A 核心）

**Day 1-3: 入职适应 + 了解团队标准**
- 完成 IT 环境搭建、账号权限开通、HR 入职流程
- 获取公司现有 2-3 个已上线 App 的完整 PRD + Figma 设计稿
- 与团队 leader 1:1 确认：PRD 格式要求、评审标准、交付节奏、团队协作流程
- 记录团队使用的术语、文档结构、关键字段
- **产出：** 一份《团队 PRD 质量标准清单》，列出 PRD 必须包含的所有字段和格式规范

**Day 4-6: 构建 PRD 模板**
- 用 Claude 分析现有 PRD，提取通用结构：
  - 产品概述模板（目标用户、核心场景、竞品分析框架）
  - 功能清单模板（扫描模块、识别结果模块、付费墙模块、设置模块）
  - 技术需求模板（AI 接口需求、订阅系统需求、数据埋点需求）
- 将模板存储为 Claude Project 的 Knowledge（Claude Projects 支持上传参考文档作为对话上下文，上传 PRD 模板后，新对话中可直接基于模板生成内容），实现一键生成
- 用《团队 PRD 质量标准清单》逐项验证模板完整性

**Day 7-8: 构建 Figma 原型模板**
- 拆解现有 App 的 Figma 文件，识别和标记可复用组件：
  - 启动页 + 引导页模板
  - 扫描页框架（相机取景器 + 扫描动画 + 结果展示）
  - 付费墙模板（周订阅定价 + 功能对比 + CTA）
  - 设置页模板
- 创建基于 Figma Frame 的页面级布局模板（非完整 Component，而是可复制粘贴的页面骨架），包括：启动页、扫描页、结果页、付费墙页、设置页各一个标准布局。完整 Component Library（含 Auto Layout、Variants、命名规范）在后续迭代中持续完善

> 注：完整的 Figma Component Library（含 Auto Layout、Variants、命名规范）需要持续迭代，不要求一次完成。

#### 第 3 周：第一个产品交付

**Day 9-10: 从产品池子挑选新 App 方向**
- 产品池子：公司维护的待开发 App 创意清单，PM 可自主选择认为有市场潜力的方向
- 用 Claude 辅助分析各方向的市场潜力（App Store 同类竞品的下载量、评分、订阅定价）
- 产出选品分析报告（这本身就是展示 AI 能力的机会）

**Day 11-12: 用模板 + AI 快速产出 PRD**
- 用 Claude + PRD 模板生成完整 PRD 初稿（预计 2-3 小时）
- 人工审核和调整，确保符合团队标准（预计半天）
- 提交 PRD 评审
- **预留 Day 13 作为评审反馈修改缓冲**

**Day 13-15: 原型设计**
- 如果 Figma Make 可用：用 Claude 将 PRD 用户故事转化为 Figma Make prompt → 生成原型初稿（预计 1-2 天）
- 如果 Figma Make 不可用：使用已构建的 Figma 页面骨架模板，手动组装原型（预计 2-3 天，仍比从零设计快得多）
- 结合模板组件库调整和完善
- **如需设计评审修改，可顺延至下周初 1-2 天**

> **关于 Figma Make：** Figma Make 是 Figma 内置的 AI 功能，可从自然语言描述直接生成高保真交互原型（支持选择 Claude 4.6 或 Gemini 3 作为底层模型）。目前已普遍可用，但输出质量取决于 prompt 质量和设计复杂度。如果输出不理想，回退到手动 Figma 设计 + 组件库复用。

#### 第 4 周：建设 AI 全栈能力（Approach B 启动）

- 总结第一个产品的完整流程，记录哪些环节 AI 加速了、哪些没有
- 搭建 Claude → Figma Make 的完整工作流（如果验证可用）
- 建立竞品分析 AI prompt 库
- 建立用户调研分析 AI prompt 库
- 用 Claude Code 辅助生成技术问题清单，与 iOS/后端开发讨论可行性（注意：技术验证由工程团队负责，PM 的角色是提出有质量的问题）
- 优化 PRD 模板和 Figma 组件库

> **如果现有 PRD 格式不统一或文档不完整：** 模板构建可延至第 2 周末完成，相应延后首个产品交付 2-3 天。这不影响整体计划——模板质量比速度更重要。

#### 第 2 个月起：流程优化和团队推广

> 注：以下为方向性目标，具体内容将在第一个月结束时基于实际经验制定详细计划。投流素材设计和数据分析属于不同职能线，暂列为探索方向而非承诺交付。

- 将个人 AI 工作流文档化，分享给团队
- 推动团队在 PRD 评审中使用 AI 辅助检查
- **本计划 Scope 外（仅记录供未来参考）：** AI 辅助投流素材设计、数据分析等属于不同职能线，不纳入本方案，如有需要另行立项
- 目标：每个新 App 从概念到 PRD+原型交付，稳定在 5 天以内

### Risks & Mitigations

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| AI 工具不被允许使用 | 中 | 高 | 降级方案：Claude 网页版辅助写作 + Figma 手动设计 + 组件库加速 |
| Figma Make 输出质量不足 | 中 | 低 | 回退到 Figma 模板组件手动组装，仍比从零设计快 50%+ |
| 第一次 PRD 评审未通过 | 高 | 中 | 已预留 1 天缓冲；主动在评审前请 leader 非正式预审 |
| 团队对 AI 工作方式有抵触 | 低 | 中 | 只展示结果不强推工具，先用交付质量建立信任 |
| 入职适应期超预期 | 中 | 中 | 路线图已预留 3 天入职适应，最差情况第一个交付延后 1 周 |

### PRD 模板大纲（实际模板需根据团队标准调整，以下为参考结构）

```
1. 产品概述
   - 产品名称 & 一句话描述
   - 目标用户画像
   - 核心价值主张
   - 竞品分析（App Store Top 10 同类 App）

2. 功能需求
   2.1 引导流程
       - 启动页（品牌展示）
       - 引导页（3-4 页功能介绍）
       - 付费墙（首次打开）
   2.2 核心功能 - 扫描识别
       - 相机扫描页（取景框 + 拍照 + 相册导入）
       - 识别处理（loading 动效）
       - 结果展示页（识别对象详情）
   2.3 内容体系
       - 识别对象数据库需求
       - 内容展示层级和字段
       - 收藏/历史记录
   2.4 订阅系统
       - 付费墙设计（周订阅/月订阅/年订阅）
       - 免费 vs 付费功能边界
       - 订阅恢复 & 取消
   2.5 辅助功能
       - 设置页
       - 用户反馈
       - 隐私政策 & 使用条款

3. 非功能需求
   - AI 识别接口选型（API 提供商 & 精度要求）
   - 性能要求（识别速度 < 3 秒）
   - 数据埋点需求（关键转化漏斗）

4. 上线 Checklist
   - App Store 审核要点
   - 投流素材需求（注：可能由增长/运营团队负责，需确认后决定是否保留在 PRD 中）
   - ASO 关键词策略（注：同上，需确认职责归属）
```

## Acceptance Criteria（PRD 质量标准 — 入职后根据团队实际标准调整）

一份 PRD 可提交评审的最低标准：
- [ ] 包含团队标准模板中的所有必填字段
- [ ] 功能需求覆盖完整用户流程（从启动到付费转化的每个步骤）
- [ ] 每个功能点有明确的验收条件
- [ ] 竞品分析包含至少 3 个 App Store 同类产品的数据
- [ ] 付费墙设计包含具体的定价方案和免费/付费边界
- [ ] 非功能需求包含 AI 接口选型建议和性能指标

一份 Figma 原型可提交评审的最低标准：
- [ ] 覆盖核心用户流程的所有页面（引导→扫描→结果→付费墙→设置）
- [ ] 页面间有可点击的交互连接
- [ ] 使用符合团队规范的设计规格（字体、颜色、间距）

## Success Criteria

## What I noticed about how you think

- 你说"先达到传统产品经理的水平"——这说明你不是在追求华而不实的 AI 噱头，而是真正理解"交付质量"是第一位的。很多人入职就急着展示新工具，结果产出的东西团队无法直接使用。

- 你对"App 工厂"模式的理解很到位——你清楚这不是做一个精品 App，而是一个快速试错的投资组合策略，速度和数量比单个产品的完美度更重要。
